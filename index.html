Fundamentos de Testing para aprender como QA profesional

Esta guía explica los conceptos esenciales de testing de software con ejemplos genéricos de una plataforma digital con asistente inteligente, interfaz web, integraciones y flujos de recomendación. El enfoque es público, educativo y reutilizable.

Objetivos del temario

La meta no es repetir definiciones: es entender cómo se razona una prueba, cómo se detecta riesgo y cómo se comunica evidencia.

Explicar qué es testing

Diferenciar testing, debugging, aseguramiento de calidad, verificación y validación.

Comprender su valor

Relacionar defectos, fallos, riesgo, costo, confianza y calidad del producto.

Aplicar pensamiento QA

Diseñar casos claros, priorizar flujos críticos, documentar evidencia y reportar defectos útiles.

Contexto de ejemplo: una plataforma genérica de recomendaciones digitales donde un usuario pide ayuda a un asistente inteligente, revisa sugerencias, ejecuta acciones en UI y recibe respuestas generadas por IA.

1.1 ¿Qué es testing?

Definición práctica: probar software es evaluar un producto o componente mediante revisión, análisis y ejecución para descubrir defectos, verificar cumplimiento de requisitos, validar necesidades del usuario y aportar información sobre riesgos.
Verificación

¿Construimos el producto correctamente conforme a especificación?

Validación

¿Construimos el producto correcto para la necesidad real del usuario?

Quality Control

Detecta defectos en el producto mediante pruebas, revisiones y medición.

Quality Assurance

Mejora el proceso para prevenir problemas y elevar consistencia.

Ejemplo aplicado

Estático: revisar que la historia indique datos requeridos, restricciones y comportamiento esperado. Dinámico: abrir la aplicación, ingresar una solicitud, observar la respuesta y comparar contra el resultado esperado.

1.2 ¿Por qué es necesario testing?

Porque el software falla, los requisitos son incompletos, los usuarios interpretan diferente y los cambios introducen riesgo.

Situación

Un usuario pide al asistente: “muestra oportunidades para mejorar mi campaña”.

Riesgo

El sistema podría usar datos desactualizados, recomendar una acción incorrecta o bloquear la UI.

Error → Defecto → Fallo

ConceptoExplicaciónEjemplo
ErrorAcción humana equivocada.Se interpreta mal la regla para calcular elegibilidad.
DefectoProblema introducido en un artefacto.La regla queda invertida.
FalloManifestación observable durante ejecución.La UI permite aplicar una recomendación que debía estar bloqueada.
Valor del testing: reduce incertidumbre, aumenta confianza y entrega información para decidir si un producto puede liberarse.

1.3 Siete principios de testing

1. Presencia de defectos

Testing muestra que hay defectos, no que no existe ninguno.

2. Exhaustivo es imposible

No se pueden probar todas las combinaciones reales.

3. Probar temprano

Revisar requisitos y diseño evita defectos costosos.

4. Defectos se agrupan

Algunos módulos concentran más fallos.

5. Paradoja del pesticida

Si repites siempre las mismas pruebas, encuentras menos defectos nuevos.

6. Depende del contexto

No se prueba igual una app bancaria que un asistente basado en IA.

7. Ausencia de errores

Sin fallos aparentes no significa que el producto sea útil.

1.5 Habilidades y buenas prácticas

Pensamiento analítico

Descompone flujos, estados, condiciones y riesgos.

Comunicación clara

Explica defectos con precisión y evidencia.

Conocimiento técnico

Comprende UI, API, datos, logs, ambientes e integración.

Casos de prueba educativos

TC-001 · Recomendación visible

Verificar que el asistente muestre una recomendación cuando existen datos válidos.

TC-002 · Acción bloqueada

Confirmar que el sistema no permita aplicar una acción cuando falta una condición.

TC-003 · Regresión crítica

Asegurar que una función existente siga operando después de un cambio.

TC-004 · Integración externa

Validar manejo de error cuando una dependencia responde tarde.

Defectos, evidencia y trazabilidad

Título: El asistente muestra como aplicable una recomendación bloqueada por regla de elegibilidad.

Actual: la acción aparece habilitada y permite continuar.

Esperado: la acción debe mostrarse bloqueada porque la cuenta no cumple la condición mínima definida.

Evidencia: video, captura, timestamp y log/traza si el error requiere análisis backend.

¿Qué es un oráculo?

¿Qué es trazabilidad?

Conexión entre requisito, caso, ejecución, evidencia y defecto.
