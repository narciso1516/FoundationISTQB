🎯 CAPÍTULO 2

¿Qué vas a aprender?

Este capítulo responde una pregunta central para la vida profesional de QA: ¿qué debo probar, en qué momento y con qué profundidad? En un proyecto real, el tester no solo recibe una pantalla terminada. También participa revisando historias, criterios de aceptación, dependencias, datos, riesgos, defectos y regresión.

🧭Ubicar QA en el SDLC

Entender cuándo aportar: antes de desarrollar, durante el sprint, en integración, antes del release y después de producción.

🧱Separar niveles de prueba

No confundir probar una función aislada con probar un flujo completo o validar que el negocio acepte el resultado.

🔁Decidir regresión

Cuando algo cambia, QA debe pensar qué podría haberse roto, no solo repetir casos por costumbre.

Ejemplo central: una plataforma con asistente inteligente recomienda acciones al usuario. QA debe validar reglas, UI, API, datos, mensajes, permisos, errores y regresión.

2.1 · SDLC

Testing en el ciclo de vida del desarrollo de software

El SDLC es el camino que sigue el software desde la idea hasta su mantenimiento. Para un tester, el SDLC es importante porque indica cuándo conviene intervenir. Si QA entra demasiado tarde, solo encuentra problemas cuando ya son más caros de corregir.

📋1. Requisitos

QA revisa si la historia es clara, si hay reglas faltantes y si los criterios son verificables.

🧩2. Diseño

QA identifica pantallas, datos, APIs, permisos, dependencias y posibles puntos de falla.

⚙️3. Desarrollo

QA prepara casos, datos, escenarios positivos, negativos y borde.

🧪4. Pruebas

QA ejecuta, compara contra oráculos, documenta evidencia y reporta defectos.

🚀5. Release

QA informa riesgo residual, resultados, bloqueos y recomendación de salida.

MomentoQué hace un tester profesionalEjemplo práctico
Antes de desarrollarCuestiona ambigüedades y pide criterios claros.“¿La recomendación debe mostrarse si la cuenta no tiene datos históricos?”
Durante el sprintPrepara datos, escenarios y revisión de impacto.Crea una cuenta elegible, una bloqueada y una sin información suficiente.
Antes del releaseEjecuta smoke, regresión crítica y valida evidencia.Confirma que el asistente responde, muestra tarjeta y respeta permisos.
Después del releaseObserva errores, feedback, métricas y defectos productivos.Revisa reportes de usuarios o logs si la recomendación falla en producción.
Ejemplo explicado como tester

Supón que un desarrollador cambia la regla que habilita una recomendación. Un tester principiante solo prueba el botón. Un tester profesional revisa la regla, los datos, el mensaje visible, la API, la UI, el caso negativo, el caso borde y la regresión de recomendaciones similares.

MODELOS

Modelos de desarrollo y cómo cambia QA

No todos los proyectos se prueban igual. El modelo de desarrollo cambia la velocidad, la documentación disponible, la frecuencia de entrega y la forma de reportar riesgo.

🏗️Secuencial

Las fases van una después de otra. QA debe insistir en revisiones tempranas porque los defectos tarde cuestan más.

Vida diaria: si te dan un documento grande, marca reglas ambiguas antes de diseñar casos.
🔄Iterativo

El producto se mejora por ciclos. QA aprende de cada iteración y ajusta pruebas.

Vida diaria: después de cada demo, actualiza tus casos con lo que cambió.
📦Incremental

Se agregan partes funcionales. QA debe cuidar que lo nuevo no rompa lo anterior.

Vida diaria: cada nueva tarjeta exige regresión de tarjetas existentes.
⚡Ágil / DevOps

QA colabora todos los días, automatiza lo crítico y revisa el pipeline.

Vida diaria: participa en refinamiento, daily, demo y retro con evidencia.
2.2 · NIVELES

Niveles de prueba: de lo pequeño a lo completo

Un nivel de prueba indica el tamaño o capa del sistema que estás evaluando. Para entenderlo fácil: primero revisas piezas pequeñas, luego cómo se conectan, después el sistema completo y al final si realmente sirve al negocio o usuario.

NivelQué significaEjemplo de la plataformaCómo piensa QA
ComponentePrueba una pieza aislada.Función que calcula si una recomendación está habilitada.“¿La regla devuelve el resultado correcto con estos datos?”
IntegraciónPrueba comunicación entre piezas.API manda blocked=true y la UI debe bloquear el botón.“¿La UI entiende correctamente lo que responde la API?”
SistemaPrueba el producto completo.Usuario abre asistente, pide recomendación, revisa tarjeta y aplica acción.“¿El flujo completo funciona de inicio a fin?”
AceptaciónValida necesidad real de usuario o negocio.El usuario entiende qué hacer y por qué la recomendación importa.“¿Esto realmente resuelve el problema del usuario?”
Analogía: si pruebas una bicicleta, componente es revisar el freno; integración es revisar que freno y llanta respondan juntos; sistema es andar en la bici; aceptación es confirmar que sirve para ir seguro al trabajo.
2.3 · TIPOS

Tipos de prueba: qué característica quieres evaluar

El tipo de prueba responde: ¿qué quiero descubrir? No es lo mismo validar una regla de negocio que evaluar rendimiento, seguridad, accesibilidad o efectos colaterales después de un cambio.

✅Funcional

Verifica que el sistema haga lo que debe hacer.

Ejemplo: al pedir una recomendación válida, debe mostrarse una tarjeta con explicación y acción disponible.
📈No funcional

Evalúa rapidez, usabilidad, compatibilidad, seguridad y accesibilidad.

Ejemplo: la tarjeta debe cargar rápido y ser legible en pantalla pequeña.
🧬Caja blanca

Usa conocimiento interno del código, condiciones, ramas o lógica.

Ejemplo: revisar que se cubran las ramas elegible, bloqueado y sin datos.
🔧Confirmación

Repite la prueba que fallaba para comprobar que el bug quedó corregido.

Ejemplo: volver al BUG-041 y verificar que ya no permite aplicar acción bloqueada.
🔁Regresión

Busca que el cambio no rompió algo que antes funcionaba.

Ejemplo: después del fix, probar también recomendaciones válidas, filtros y historial.
🧭Exploratoria

Aprendes, diseñas y ejecutas al mismo tiempo para encontrar riesgos no obvios.

Ejemplo: probar prompts raros, datos incompletos y navegación inesperada.
2.4 · MANTENIMIENTO

Testing de mantenimiento: cuando el software ya existe

El mantenimiento ocurre cuando el sistema ya está vivo y recibe cambios. Para un tester, esta etapa es crítica porque cada cambio puede afectar funciones que antes funcionaban. Aquí aparece una habilidad muy profesional: análisis de impacto.

📝1. Qué cambió

Identifica archivo, módulo, regla, pantalla, API o dato modificado.

🕸️2. Qué depende

Busca flujos conectados, permisos, datos e integraciones.

⚠️3. Qué riesgo hay

Prioriza lo crítico para usuario, negocio y release.

🧪4. Qué probar

Selecciona confirmación, regresión, integración o sistema.

📸5. Evidencia

Guarda capturas, video, logs, datos, build y resultado.

Vida diaria de QA: si te dicen “solo fue un cambio pequeño”, no lo aceptes sin analizar. Pregunta: ¿qué tocó?, ¿qué consume eso?, ¿qué usuarios afecta?, ¿qué casos debo repetir?
CambioRiesgo ocultoPrueba inteligente
Fix en botón bloqueado.Botones válidos podrían quedar bloqueados.Confirmación del bug + regresión de botones válidos.
Migración de datos.Recomendaciones pueden calcularse con datos incompletos.Pruebas de datos + sistema + smoke.
Actualización de navegador.Layout, scroll o clicks pueden fallar.Compatibilidad visual + flujo crítico.
QA REAL

Cómo se ve este capítulo en la vida diaria de un tester

Cuando trabajas en regresión, validación de features o testing manual/automatizado, este capítulo se convierte en una forma de pensar. No es teoría aislada: es tu checklist mental.

🕘Daily / refinamiento

Escuchas la historia y detectas riesgos: datos, permisos, integraciones, criterios faltantes.

Pregunta QA: “¿Cómo sabremos que esto está correcto?”
🧪Ejecución

No solo sigues pasos. Comparas actual vs esperado, capturas evidencia y observas comportamiento colateral.

Pregunta QA: “¿Esto falló por UI, datos, API o regla?”
📣Reporte

Comunicas impacto, reproducibilidad, severidad, entorno, datos y evidencia.

Pregunta QA: “¿Con esto el dev puede reproducir sin preguntarme?”

CASOS CAP 2

Casos educativos aplicados

TC-C2-001 · Componente

Objetivo: validar la función que calcula elegibilidad. Datos: cuenta con presupuesto menor al mínimo. Esperado: devuelve bloqueado. Aprendizaje: una regla pequeña puede romper todo el flujo.

TC-C2-002 · Integración

Objetivo: confirmar que UI interpreta la respuesta de API. Actual: API responde blocked=true. Esperado: botón deshabilitado y mensaje explicativo. Aprendizaje: muchas fallas nacen entre sistemas.

TC-C2-003 · Sistema

Objetivo: ejecutar el flujo completo. Pasos: abrir asistente, pedir recomendación, revisar detalle, aplicar acción. Esperado: navegación coherente y resultado correcto.

TC-C2-004 · Regresión

Objetivo: verificar que un fix no rompió funcionalidades existentes. Alcance: recomendación válida, bloqueada, historial, filtros y permisos. Aprendizaje: regresión es pensar en impacto.

FICHAS

Fichas de repaso

¿Qué es el SDLC para QA?

Es el mapa para saber cuándo participar: requisitos, diseño, desarrollo, pruebas, release y mantenimiento.

¿Confirmación y regresión son lo mismo?

No. Confirmación verifica un bug corregido. Regresión busca efectos colaterales en cosas que antes funcionaban.

¿Por qué importa integración?

Porque muchos defectos aparecen cuando UI, API, datos y reglas se conectan.

EVALUACIÓN

Evaluación rápida

1. Si la API devuelve blocked=true pero la UI habilita el botón, ¿qué nivel detecta mejor el problema?
2. Después de corregir un defecto, ¿qué combinación es más profesional?
3. ¿Qué pregunta representa mejor el análisis de impacto?
