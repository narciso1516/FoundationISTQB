Testing durante el ciclo de vida del desarrollo de software

Este capítulo explica cómo se integra el testing en el SDLC, cómo cambia el enfoque según el modelo de desarrollo, cuáles son los niveles de prueba y cómo se prueban cambios de mantenimiento en una plataforma digital con asistente inteligente.

Objetivos del capítulo

La meta es entender en qué momento se prueba, qué se prueba y por qué cambia la estrategia según el contexto del proyecto.

Relacionar testing y SDLC

Comprender que testing no ocurre solo al final; acompaña análisis, diseño, implementación, entrega y mantenimiento.

Distinguir niveles de prueba

Componente, integración, sistema y aceptación: cada nivel tiene objetivos, bases de prueba y defectos típicos.

Aplicar pensamiento de regresión

Analizar impacto de cambios para decidir qué volver a probar y qué evidencia documentar.

Contexto de ejemplo: una plataforma genérica de recomendaciones digitales con UI web, motor de reglas, API, datos históricos y un asistente inteligente que genera sugerencias.

2.1 Testing y SDLC

Idea central: el testing debe adaptarse al ciclo de vida del desarrollo. No es una fase aislada; es una actividad continua de verificación, validación, análisis de riesgo y generación de información.
Etapa del SDLCActividad de testingEjemplo aplicado
RequisitosRevisar ambigüedades, reglas faltantes y criterios de aceptación.La historia debe indicar cuándo una recomendación se considera elegible.
DiseñoAnalizar arquitectura, interfaces y riesgos técnicos.Validar cómo UI, API y motor de reglas intercambian datos.
ImplementaciónDiseñar casos, preparar datos, automatizar checks y ejecutar pruebas tempranas.Crear datos para cuentas elegibles y no elegibles.
EntregaEjecutar smoke, regresión, pruebas críticas y evidencia de liberación.Confirmar que el flujo principal del asistente sigue funcionando.
MantenimientoAnalizar impacto, probar correcciones y ejecutar regresión selectiva.Un cambio en reglas de elegibilidad requiere volver a probar recomendaciones relacionadas.
Ejemplo de pregunta tipo examen

Pregunta: ¿Por qué conviene involucrar testing desde requisitos? Respuesta: porque permite detectar defectos temprano, reducir retrabajo y diseñar pruebas alineadas al riesgo.

2.2 Modelos de desarrollo

El modelo de desarrollo determina el ritmo, la documentación disponible, la frecuencia de cambios y la forma de planear pruebas.

Secuencial

Fases ordenadas. Útil cuando el alcance es estable, pero los defectos tardíos pueden ser costosos.

Iterativo / incremental

El producto se construye por incrementos. Permite retroalimentación y regresión frecuente.

Ágil

Testing se integra en el equipo. Hay colaboración continua, pruebas tempranas y entrega frecuente.

Comparación práctica

ModeloRiesgo para QABuena práctica
SecuencialDetectar defectos tarde si QA entra hasta el final.Revisiones tempranas de requisitos y diseño.
IncrementalRomper funciones previas al agregar nuevas capacidades.Regresión por incremento y trazabilidad.
ÁgilCambios rápidos con documentación ligera.Criterios de aceptación claros, Definition of Done y automatización útil.
DevOps / entrega continuaLiberaciones frecuentes con alto riesgo de regresión.Pipeline, smoke automatizado, monitoreo y rollback.

2.3 Niveles de prueba

Un nivel de prueba agrupa objetivos, objetos de prueba, bases de prueba, defectos típicos y responsabilidades.

Componente

Prueba unidades o módulos aislados.

Integración

Prueba interfaces entre componentes o sistemas.

Sistema

Prueba el sistema completo contra requisitos.

Aceptación

Confirma si el producto satisface necesidades de negocio o usuario.

NivelObjeto de pruebaDefecto típicoEjemplo
ComponenteFunción de cálculo de elegibilidad.Condición mal implementada.La función devuelve true cuando debería devolver false.
IntegraciónUI + API + motor de reglas.Contrato o formato incorrecto.La API devuelve status “blocked”, pero la UI lo interpreta como “enabled”.
SistemaPlataforma completa.Flujo end-to-end incorrecto.El usuario aplica una recomendación inválida desde la pantalla final.
AceptaciónProducto desde perspectiva de negocio.No satisface necesidad real.La recomendación es técnicamente válida, pero no aporta valor al usuario.

2.4 Tipos de prueba

Los tipos de prueba responden a la pregunta: ¿qué característica de calidad estamos evaluando?

Funcionales

Validan lo que el sistema debe hacer: reglas, flujos, cálculos, mensajes y permisos.

No funcionales

Evalúan rendimiento, usabilidad, accesibilidad, seguridad, compatibilidad y confiabilidad.

Caja blanca

Evalúan estructura interna, cobertura, ramas, condiciones o caminos del código.

Pruebas relacionadas con cambio

TipoObjetivoEjemplo
Confirmación / re-testingVerificar que el defecto corregido ya no ocurre.Repetir el caso que fallaba cuando la recomendación aparecía habilitada.
RegresiónVerificar que el cambio no rompió funciones existentes.Probar recomendaciones, historial, filtros, permisos y pantalla de resumen.
Clave ISTQB: re-testing confirma una corrección específica; regresión busca efectos colaterales en partes que antes funcionaban.

2.5 Testing de mantenimiento

El mantenimiento ocurre cuando el software ya existe y recibe cambios: correcciones, mejoras, migraciones, cambios de ambiente o retiro de funciones.

Trigger de mantenimiento

Cambio de regla de negocio, actualización de API, migración de base de datos, parche de seguridad o retiro de una pantalla.

Riesgo principal

Impactar funciones existentes, datos históricos, integraciones, rendimiento o experiencia del usuario.

Análisis de impacto

PreguntaUso para QA
¿Qué cambió?Identifica componente, requisito, regla, API o UI modificada.
¿Qué depende de eso?Detecta flujos, datos, permisos e integraciones afectadas.
¿Qué casos deben repetirse?Define regresión selectiva y prioridades.
¿Qué evidencia se necesita?Prepara capturas, videos, logs, datos y trazabilidad.

Shift-left, CI/CD y DevOps

Shift-left significa mover actividades de calidad hacia etapas más tempranas. En vez de esperar al final, QA participa desde requisitos, diseño, refinamiento, revisiones y automatización temprana.

Shift-left

Detecta problemas antes de construir o liberar.

CI/CD

Ejecuta validaciones frecuentes en cada cambio.

DevOps

Integra desarrollo, pruebas, operación, monitoreo y mejora continua.

Aplicación práctica: en una regresión semanal, QA puede combinar checklist manual crítico, pruebas automatizadas de smoke, revisión de logs y evidencia visual de los flujos principales.

Casos educativos del Capítulo 2

TC2-001 · Prueba de componente

Validar que la función de elegibilidad devuelva “bloqueado” cuando la cuenta no cumple la condición mínima.

TC2-002 · Prueba de integración

Confirmar que la UI interprete correctamente el estado devuelto por la API.

TC2-003 · Prueba de sistema

Ejecutar el flujo completo: solicitar recomendación, revisar tarjeta, aplicar acción y validar resultado final.

TC2-004 · Regresión de mantenimiento

Después de cambiar una regla, repetir flujos críticos para verificar que no se afectaron recomendaciones previas.
