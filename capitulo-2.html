Testing durante el ciclo de vida del desarrollo de software

Este capítulo explica cómo el testing cambia según el modelo de desarrollo, el nivel de prueba, el tipo de prueba y el mantenimiento del producto. El enfoque usa ejemplos genéricos de una plataforma digital con asistente inteligente, interfaz web, integraciones y flujos de recomendación.

Objetivos del capítulo

La meta es entender dónde se ubica el testing dentro del desarrollo de software y cómo elegir el enfoque correcto según el contexto.

Relacionar testing y SDLC

Comprender cómo cambia el testing en modelos secuenciales, iterativos, incrementales, ágiles y DevOps.

Distinguir niveles de prueba

Identificar pruebas de componente, integración, sistema y aceptación.

Diferenciar tipos de prueba

Separar pruebas funcionales, no funcionales, caja blanca, relacionadas al cambio y regresión.

Idea clave: no se prueba igual un componente aislado que una integración, una funcionalidad completa o una validación de negocio con usuarios.

2.1 Testing en el ciclo de vida del desarrollo de software

Definición práctica: el testing debe integrarse desde etapas tempranas del ciclo de vida, no únicamente al final. Revisar requisitos, analizar riesgos y diseñar pruebas temprano reduce retrabajo y defectos costosos.
ModeloCómo se pruebaRiesgo si se prueba tarde
SecuencialLas fases avanzan en orden: requisitos, diseño, construcción, pruebas y liberación.Los defectos de requisitos aparecen cuando ya son costosos.
Iterativo / incrementalSe prueba por incrementos funcionales y se obtiene retroalimentación frecuente.Puede acumular deuda de regresión si no hay cobertura estable.
ÁgilQA participa en refinamiento, criterios de aceptación, pruebas por historia y regresión continua.Historias rápidas con criterios débiles generan ambigüedad.
DevOpsEl testing se automatiza dentro del pipeline y se monitorea después de liberar.Una entrega rápida sin observabilidad puede ocultar fallos productivos.
Ejemplo aplicado

Si la plataforma agrega una nueva tarjeta de recomendación, QA no espera hasta producción. Revisa la historia, define criterios de aceptación, prepara datos, prueba UI, valida integración y ejecuta regresión sobre recomendaciones existentes.

2.2 Niveles de prueba

Un nivel de prueba agrupa objetivos, bases de prueba, objetos de prueba, defectos típicos y responsables.

Componente

Prueba unidades aisladas: función, clase, módulo o componente UI.

Integración

Prueba interfaces entre componentes, servicios, APIs o sistemas externos.

Sistema

Prueba el sistema completo contra requisitos funcionales y no funcionales.

Aceptación

Valida si el producto satisface necesidades de negocio, usuario o contrato.

NivelObjeto de pruebaDefecto típicoEjemplo QA
ComponenteFunción que calcula elegibilidad.Condición mal implementada.Validar que una cuenta no elegible devuelva estado bloqueado.
IntegraciónUI + API de recomendaciones.Mapeo incorrecto de campos.La API devuelve blocked=true, pero la UI muestra botón activo.
SistemaFlujo completo de recomendación.Fallo en navegación o reglas completas.Crear solicitud, recibir recomendación, revisar detalle y aplicar acción.
AceptaciónProceso de negocio.El flujo no resuelve la necesidad real.El usuario entiende por qué una acción está bloqueada y qué hacer después.

2.3 Tipos de prueba

El tipo de prueba expresa el objetivo de la evaluación. Puede aplicarse en varios niveles.

Funcional

Verifica qué hace el sistema respecto a reglas, flujos y resultados esperados.

No funcional

Evalúa atributos como rendimiento, usabilidad, seguridad, compatibilidad y accesibilidad.

Caja blanca

Diseña pruebas con conocimiento de estructura interna, código, ramas o condiciones.

Relacionado al cambio

Confirma que un defecto fue corregido o que un cambio no rompió algo existente.

Confirmación

Reejecuta una prueba fallida para comprobar que el bug quedó corregido.

Regresión

Busca efectos colaterales en funcionalidades que antes funcionaban.

Diferencia crítica: confirmación vs regresión

ConceptoPregunta que respondeEjemplo
Prueba de confirmación¿El defecto reportado ya fue corregido?BUG-041: la recomendación bloqueada ya no permite aplicar acción.
Prueba de regresión¿El cambio rompió algo que antes funcionaba?Después del fix, las recomendaciones válidas siguen mostrándose y aplicándose.

2.4 Testing de mantenimiento

El mantenimiento ocurre cuando un sistema ya existe y recibe cambios: correcciones, mejoras, migraciones, actualizaciones de ambiente, cambios regulatorios o retiro de funciones.

Disparadores de mantenimiento

Bug fix, nueva feature, refactor, migración de API, cambio de base de datos, actualización de navegador, ajuste de seguridad o retiro de módulo.

Impacto QA

Analizar impacto, seleccionar regresión, validar integración, verificar datos, confirmar corrección y documentar evidencia.

CambioRiesgoPrueba necesaria
Se actualiza el servicio de recomendaciones.La UI interpreta mal una respuesta nueva.Integración + regresión de flujos críticos.
Se corrige un bug de elegibilidad.La corrección afecta recomendaciones válidas.Confirmación + regresión funcional.
Se migra la base de datos.Datos incompletos o inconsistentes.Pruebas de datos, sistema y smoke test.

Casos de prueba educativos

TC-C2-001 · Componente

Validar que el cálculo de elegibilidad devuelva “bloqueado” cuando faltan condiciones mínimas.

TC-C2-002 · Integración

Confirmar que la UI respete el campo blocked recibido desde la API.

TC-C2-003 · Sistema

Ejecutar el flujo completo: solicitud al asistente, recomendación, detalle y acción.

TC-C2-004 · Aceptación

Verificar que el usuario comprenda la recomendación, la razón y el siguiente paso.

Matriz de decisión rápida

Si el cambio toca...Prioriza...Evidencia mínima
Reglas de negocioComponente + sistema + regresiónDatos usados, resultado esperado, captura y log.
Contrato APIIntegración + sistemaRequest, response, timestamp y captura UI.
Diseño visualSistema + no funcional de usabilidad/accesibilidadCapturas comparativas y navegador usado.
Corrección de bugConfirmación + regresión selectivaID del bug, build, resultado y evidencia.
Regla profesional: no ejecutes toda la regresión por costumbre; selecciona pruebas según impacto, riesgo, historial de defectos y criticidad del flujo.

Flashcards de repaso

¿Qué define un nivel de prueba?

El objetivo, objeto de prueba, base de prueba, defectos típicos y responsables en una capa del sistema.

¿Confirmación y regresión son lo mismo?

No. Confirmación verifica el bug corregido; regresión busca efectos colaterales.

¿Por qué probar temprano?

Porque detectar ambigüedad en requisitos cuesta menos que corregir defectos al final.

Evaluación rápida

1. ¿Cuál es un ejemplo de prueba de integración?
2. Después de corregir un bug, ¿qué combinación es más adecuada?
3. ¿Qué nivel valida el sistema completo contra requisitos?
